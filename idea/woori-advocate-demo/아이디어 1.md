# 내편 AI (Woori Advocate) — 팀원 설명서

---

## 1. 우리가 만드는 것

> **같은 LLM을 두 페르소나로 격리시켜, "Seller(은행 편)"가 상품을 팔 때 "Advocate(고객 편)"가 같은 상품의 위험을 좌우 분할 화면에 동시에 보여주는 시스템.**
> 

**슬로건(메인):** "우리은행은 고객 편을 드는 AI를 둡니다."
**슬로건(서브):** "세계 최초, 자기 행원에게 반대하는 AI."

### 작동 채널 2가지

**(1) 비대면 채널 — 우리WON뱅킹 앱 (B2C)**

- 고객이 앱에서 고위험 상품 가입 버튼을 누르는 순간 화면이 일시 비활성화
- 좌측: Seller (상품 강점·예상 수익)
- 우측: Advocate (부적합 이유·위험 요인)
- 고객이 Advocate 경고 항목을 하나씩 "이해함" 체크해야만 가입 진행 가능

**(2) 대면 채널 — 영업점 창구 (B2B2C)**

- 행원 PC 모니터: Seller 관점 (셀링 포인트·설명 스크립트 가이드)
- 고객 앞 태블릿: Advocate 관점 (개인 맞춤 리스크 경고)
- 창구 마이크 STT로 상담 내용 실시간 텍스트화 → 양쪽 화면 동기

### 기존 은행 AI와 무엇이 다른가

| 비교 축 | 기존 모든 은행 AI | 우리 Advocate |
| --- | --- | --- |
| **관점** | 상품을 더 잘 팔기 위한 AI | **고객 편에서 반대하는 AI** |
| **시점** | 사후 (해피콜·녹취 분석) | **사전 (가입 직전 개입)** |
| **출력** | 단일 관점 추천 | **이중 관점 동시 출력** |
| **근거** | 체크리스트·키워드 | **RAG 기반 약관 맥락 분석** |
| **설명** | "이렇게 사세요" | **"이게 당신에게 안 맞는 이유"** |

신한·KB·하나·JP Morgan·Bank of America — 전부 "더 잘 팔기 위한" 은행 편 AI다.
**"고객 편" AI는 상용 적용 사례가 없다. 이게 우리가 노리는 포지션.**

### 핵심 역전 3가지 (발표 시 강조 포인트)

1. **관점의 역전** — 모든 경쟁사 AI는 은행 편. Advocate만 고객 편.
2. **시점의 역전** — 사후 해피콜·녹취가 아닌 가입 직전 사전 개입.
3. **출력의 역전** — 단일 관점 추천이 아닌 이중 관점 동시 출력.

## 2. 왜 만드나 (배경)

### 우리은행의 진짜 약점 3가지

**① DLF 불완전판매 사태 (2019)**

- 우리은행 과태료 **197.1억원** — 은행권 역대 최대
- 사모펀드 신규판매 **6개월 정지**, 손태승 은행장 **중징계(문책경고)**
- 분쟁조정 평균 배상비율 55% 결정
- 이 사태는 "고객 프로필에 맞지 않는 고위험 상품을 아무 경고 없이 팔았기 때문"
- **Advocate가 2019년에 있었다면 부적합 고객에게 가입 직전 경고 → 사태 자체가 없었음**

**② 소비자보호 실태평가 '보통' (2025.12 금감원 발표)**

- 29개 금융사 중 '우수' 0곳, '양호' 2곳, '보통' 19곳 (우리은행 포함)
- 실태평가 핵심 항목 = 설명의무·적합성·민원관리 → **Advocate가 직접 개선하는 항목들**
- 금소법 과징금 75% 감경 조건: 자율배상 + **실태평가 우수** → Advocate 도입이 감경 근거

**③ 경쟁사 AI는 전부 "은행 편"**

| 은행 | AI 서비스 | 포지션 |
| --- | --- | --- |
| 신한 | 완전판매 AI 스크립트·AI 해피콜 | 판매 절차 사후 확인 |
| KB | 케이봇쌤·AI 금융비서 | 상품 추천·포트폴리오 |
| 하나 | 소비자보호그룹장 부행장 격상 | 조직 강화 (시스템 없음) |
| 신영증권 | AIVS (녹취 분석) | 사후 모니터링 |

→ 경쟁사는 "판매 보조 AI" 또는 조직 강화. **"고객 편 AI"로 포지셔닝한 곳은 없다.**

### 우리은행이 이미 갖고 있는 것 (개발 비용 거의 0)

| 보유 자산 | Advocate 활용 방법 |
| --- | --- |
| **우리GPT** (2024.12, RAG+LLM, 1,000만건 내부 데이터) | Seller·Advocate 베이스 LLM |
| **운영GPT** (2025.05, 리스크 특화) | 리스크 분석 로직 재활용 |
| **175개 AI 에이전트** (2026.03 배치) | Advocate = 176번째 에이전트로 추가 |
| **자체 GPU 서버 클러스터** (2026.04 구축) | 온프레미스 LLM 추론 인프라 |
| **영업점 창구 태블릿 + STT/AICC** | 대면 채널 즉시 재활용 |

→ 외부 인프라 투자 거의 없이 도입 가능. 이게 **"실현가능성 30점"을 따는 핵심 논리**다.

### 2026년이 도입 타이밍인 이유

| 사건 | 시점 | 우리에게 미치는 영향 |
| --- | --- | --- |

| EU AI Act 본격 집행 | 2026.08 | 금융 AI 투명성·인간감독 의무, 위반 시 매출 7% |
| 금소법 과징금 체계 강화 | 2025.11 개정 | 위반 시 1~100% 과징금, 감경 조건 = 실태평가 우수 |
| 우리금융 'AI 경영' 원년 선언 | 2026 | AX 추진위원회 출범, 소비자보호 전담 부서 신설 |

**2026년은 "AI 사전 컴플라이언스를 도입 안 한 은행이 차별적으로 손해 보는" 분기점.**
Advocate는 이 분기점의 첫 답변이다.

## 3. 핵심 메커니즘

### 전체 흐름

```
                    [고객이 ELS 가입 시도]
                            ↓
        ┌───────────────────┴───────────────────┐
        ↓                                       ↓
┌─────────────────┐                  ┌─────────────────┐
│   Seller LLM    │                  │  Advocate LLM   │
│ 상품 강점·수익률 │                  │ 고객 리스크·부적합 │
│ 입력: 설명서·시장 │                  │ 입력: 약관·고객 프로필 │
└────────┬────────┘                  └────────┬────────┘
         │                                    │
         └─────────────┬──────────────────────┘
                       ↓
              ┌────────────────┐
              │   Moderator    │ ← 금소법 검증·환각 차단·PII 필터
              └────────┬───────┘
                       ↓
              ┌────────────────┐
              │  좌우 분할 출력  │
              │ + 항목별 체크박스│
              └────────────────┘
```

### 각 에이전트 역할 상세

**Seller LLM**

- 입력: 상품 설명서, 시장 전망, 세제 혜택, 고객 투자 성향 결과
- 출력: 상품 핵심 강점 / 고객에게 유리한 점 3가지 / 기대 수익 시뮬레이션
- 톤: 정중·설득·글머리 기호 구조
- 제약: 단정적 표현("반드시", "확실히") 금지 (금소법 §21 부당권유 금지)

**Advocate LLM**

- 입력: 약관 전문, 위험 공시, 고객 프로필(연령·자산·소득·거래이력·최근 지출)
- 출력: "이 상품이 맞지 않을 수 있는 N가지 이유" (최대 5가지, 순번 명시)
- 톤: 비판적·팩트 기반·평어·구체적 수치
- 제약: 환각 절대 금지 → 약관·고객 데이터에 명시된 사실만 인용

**Moderator (Safety Filter)**

- Seller 출력이 부당권유 금지 위반인지 체크
- Advocate 출력에 환각(없는 정보 인용) 있는지 체크
- 두 출력에 개인정보(PII) 노출 있는지 필터
- 3회 연속 실패 시 → 백업 정적 모형(룰 기반)으로 전환

### "같은 LLM인데 왜 다른 결과가 나오나" — 자주 받는 질문 선제 이해

핵심은 **입력을 완전히 다르게** 준다는 것이다.

```python
# Seller 호출
seller_response = llm.call(
    system="상품의 강점을 설득력 있게 어필하라. [Seller 헌법]",
    context=product_summary + market_outlook  # 설명서·시장만
)

# Advocate 호출 (메모리 공유 없음)
advocate_response = llm.call(
    system="고객의 위험을 팩트로만 짚어라. [금소법 헌법]",
    context=product_terms + customer_profile  # 약관·고객 프로필만
)
```

같은 사람이라도 "홍보팀 입장에서 써달라"와 "소비자 권익팀 입장에서 써달라"는 완전히 다른 글이 나온다. 그게 프롬프트 격리의 원리.

### RAG 동작 방식 (환각 방지 핵심)

```
[상품명 입력]
    ↓
[Chroma 벡터DB에서 약관·공시 관련 청크 검색]
    ↓
[검색된 텍스트를 Advocate 프롬프트에 주입]
    ↓
[Advocate는 주입된 텍스트 범위 내에서만 위험 인용]
    ↓
[Moderator가 인용 출처 대조 검증]
```

Advocate가 "원금 손실 가능"이라고 쓰면 Moderator가 약관 원문에 해당 문구가 있는지 확인한다. 없으면 차단·재생성.

### 학술 근거 (발표 시 신뢰성 보강)

- **MIT MAD (ICML 2024):** 여러 LLM이 토론하면 단일 LLM 대비 사실 정확도 향상, 환각 감소
- **Google DeepMind (2025):** 멀티에이전트는 금융 추론 병렬 작업에서 최대 **+80.9% 성능 향상**
- **Anthropic Constitutional AI:** AI 출력을 헌법(규칙) 원칙으로 자가 검증·수정 — Advocate 시스템 프롬프트에 금소법 6대 의무를 "헌법"으로 삽입
- **Kahneman "Thinking Fast and Slow":** Seller = System 1(빠른 직관), Advocate = System 2(느린 분석) → 고위험 의사결정에서 System 2 개입이 오류율 감소

## 4. 데모 시나리오

> **시연 우선순위:** 시나리오 1이 "Seller vs Advocate 이중 관점"이라는 핵심 컨셉을 가장 잘 보여준다. 1번만 완벽하면 발표 성공. 2·3은 1분 내 빠르게 시연하거나 백업 영상으로 대체.
> 

---

### 시나리오 1. 영업점 ELS 가입 — **메인 시연 (1.5분)**

**상황:** 박영희(62세, 최근 의료비 1,200만원 지출, 5년간 고위험 상품 가입 이력 없음)가 영업점 창구에서 홍콩H지수 연계 ELS(만기 3년, 낙인 50%) 가입 시도.

**기술 연동:** 창구 마이크 STT → 실시간 텍스트화 → 양쪽 화면 동기화

**행원 PC 모니터 (Seller 관점):**

```
[행원 안내 가이드] 박영희 고객님 맞춤 세일즈 팁

▸ 금리 메리트: 시중 예적금 대비 3배 수준인 연 최대 6.0% 세전 쿠폰 금리 강조
▸ 투자 성향: 설문 결과 '적극투자형' — 본 ELS 가입 요건 충족
▸ 설명의무: 약관의 원금 손실 조건(H지수 -50% 이하 하락 시) 표준 스크립트 낭독 필수
▸ 체크포인트: 만기·중도해지 조건·과거 수익률 단계별 클릭 동의 획득
```

**고객 태블릿 화면 (Advocate 관점):**

```
"박영희 님, 가입 서명 전에 이 상품이 맞지 않을 수 있는 5가지 이유를 확인해 주세요."

1. 3년간 자금 묶임 (유동성 제약)
   본 상품은 3년간 중도 인출이 어렵습니다.
   박영희 님의 지난달 의료비 1,200만 원 지출을 고려할 때
   예상치 못한 병원비 비상금 마련에 차질이 생길 수 있습니다.

2. 실질 투자 경험 부재
   설문상 '적극투자형'이지만, 우리은행 거래 기록 기준
   최근 5년간 원금 비보장형 고위험 상품 가입 이력이 전혀 없습니다.

3. 원금 손실 가능 조건
   만기 시 홍콩 지수가 가입일 대비 -50% 이상 하락 시
   원금 최대 100% 손실 발생 가능.

4. 수익률 낙관 경고
   제시된 연 6% 금리는 지수가 일정 기준 밑으로 내려가지 않는 조건부 혜택.
   조기 상환 실패 시 기회비용 손실 증가.

5. 동일 상품군 손실 지표
   작년 유사 조건 ELS 가입 고객 중 7.2%가 만기 시 원금 손실 경험.

[ ] 위 1번 위험을 읽고 이해했습니다.
[ ] 위 2번 위험을 읽고 이해했습니다.
[ ] 위 3번 위험을 읽고 이해했습니다.
[ ] 위 4번 위험을 읽고 이해했습니다.
[ ] 위 5번 위험을 읽고 이해했습니다.

→ 모두 체크 + 자필 서명 → 가입 진행
→ 1개라도 미체크 → 행원에게 추가 설명 요청 자동 전송
```

**이 시나리오가 핵심인 이유:** DLF 사태가 정확히 이런 상황이었다. 62세 고객에게 부적합한 고위험 상품을 아무 경고 없이 판 것.

### 시나리오 2. 시니어 거액 송금 — **보조 시연 (1분)**

**상황:** 이순자(75세)가 손주 명의로 신규 개설된 계좌에 1,500만원 송금 시도.

**핵심 기술:** 거래 패턴 이상 감지 + Family Consent Vault(가족 동의 앱) 연계

**좌측 (일반 송금 가이드):**

```
"가족 송금 절차 진행 중"
▸ 수취인: 이주민 (손주 명의 계좌)
▸ 송금액: 15,000,000원
▸ 수수료 0원
```

**우측 (Advocate — 시니어 자산 방어):**

```
"이순자 님, 소중한 자금을 보내시기 전 한 번만 확인해 주세요."

1. 최근 개설 계좌 — 단 3일 전 신규 개설된 이상 계좌
2. 통화 이력 부재 — 최근 6개월 해당 손주와 통화·문자 기록 없음
3. 고령 타겟 범죄 주의 — 2026년 1분기 70대+ 여성 피해 전년 대비 +180%

[ 실제 손주 번호로 직접 확인 전화 ]
[ 자녀 앱으로 실시간 이체 동의 요청 ]

→ 동의 완료 전까지 이체 대기
→ 30분 미응답 시 우리은행 안심센터 자동 연결
```

**발표 포인트:** 시니어 모드에서는 좌우 분할 대신 한 번에 한 관점씩 보여주고 TTS가 경고를 음성으로 읽어줌. "포용금융" 키워드와 연결.

### 시나리오 3. 보이스피싱 차단 — **보조 시연 (1분)**

**상황:** 김철수(45세)가 모르는 계좌로 2,000만원 이체 시도. 스마트폰에서 원격제어 앱 감지 + 검경 사칭 의심 번호와 30분 통화 중.

**좌측 (정상 이체 창):**

```
▸ 수취인: 이영호 (신규 등록 계좌)
▸ 이체 금액: 20,000,000원
▸ 위험도 분석: 진행 가능
```

**우측 (Advocate — 금융사기 방어):**

```
"잠깐만요! 이 송금은 피싱 사기 징후를 보이고 있습니다."

1. 비정상적 거액 — 평소 평균 이체액(30만원) 대비 약 66배
2. 원격 제어 앱 동작 — 백그라운드에서 외부 원격 제어 프로그램 실행 중
3. 이상 통화 흐름 — 이체 직전 30분 이상 검경 사칭 의심 번호와 연속 통화

[ 즉시 이체 차단 및 112 신고 ]
[ 우리은행 보이스피싱 안심센터 즉시 통화 ]
```

**발표 포인트:** 기존 FDS(이상거래탐지)는 패턴 룰 기반. Advocate는 패턴 + LLM 맥락 분석 통합. 신한금융 통합 FDS가 2주에 8억 예방한 사례를 인용하며 "Advocate는 사기 차단에 더해 적합성·고령자 보호까지 통합한 확장형"으로 포지셔닝.

## 9. 심사위원이 던질 Top 5 질문 (반드시 외우기)

### Q1. "은행이 자기 상품 안 팔리게 방해하는 셈인데, 지점 KPI 타격은?"

> "고위험 상품 1건 무리 유치 이익보다, 불완전판매 민원으로 인한 지점 감점·배상 청구 리스크가 정량적으로 약 20배 큽니다. Advocate는 부적합 고객을 사전에 걸러 지점 최대 감점 요인을 원천 방어합니다. 행동경제학적으로도 '건강한 마찰(Healthy Friction)'은 적합성 원칙 충족의 필수 과정입니다."
> 

### Q2. "고객이 Advocate 보고도 가입해서 손실 나면 책임은?"

> "Advocate는 직원 책임을 갈음하지 않습니다. 다만 고객이 5가지 리스크를 항목별로 체크하고 서명한 기록이 남기에, 분쟁 시 '은행이 형식 서류를 넘어 AI 변호인까지 두어 인지시켰음'을 입증하는 결정적 Safe Harbor가 됩니다."
> 

### Q3. "병원비 결제 내역까지 활용 = 개인정보 위반?"

> "새로운 수집 없습니다. 우리은행 앱에 이미 적재된 자사 거래 내역만 온프레미스 우리GPT로 연산. 이용약관 내 '금융상품 적합성 분석 목적' 활용 동의가 이미 포함되어 있고, Advocate 호출 시 '본 분석에 거래 이력이 사용됩니다' 고지 팝업도 제공됩니다."
> 

### Q4. "환각으로 멀쩡한 가입을 막으면?"

> "RAG Guardrail로 검증된 약관·공시 텍스트 내에서만 위험을 도출. Moderator 에이전트가 자기 일관성 검증으로 오인 탐지율 1% 미만 통제. 또한 Advocate는 차단 장치가 아닌 조언자이며 최종 결정권은 고객에게 있습니다."
> 

### Q5. "동일 백엔드 LLM이면 진짜 대립 가능?"

> "프롬프트 격리 + 컨텍스트 윈도우 분리로 독립성 보장. Seller는 상품 설명서·강점만, Advocate는 약관·고객 프로필만 입력. 인간이 머릿속에서 찬반론 숙고하는 Dual-System 메커니즘과 동일 원리입니다."
>
